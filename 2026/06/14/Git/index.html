2.2、版本控制器的方式

a、集中式版本控制工具

集中式版本控制工具，版本库是集中存放在中央服务器的，团队里每个人工作时从中央服务器下载代码，必须联网才能工作（局域网或互联网），个人修改后提交到中央版本库。

  • 举例：SVN和CVS

b、分布式版本控制工具

分布式版本控制系统没有“中央服务器”，每个人的电脑上都是一个完整的版本库，工作时无需联网。多人协作只需要各自将修改推送给对方，就能互相看到对方的修改。

  • 举例：Git

2.4、Git

Git是分布式的，不需要中心服务器，每台电脑拥有的内容都是一致的。我们使用中心服务器仅仅是为了方便交换大家的修改，服务器的地位和每个人的PC相同，只是作为不关机的交换节点存在。

git是一个开源的分布式版本控制系统，可以高效、高速地处理从很小到非常大的项目版本管理。Git是 Linus Torvalds 为了帮助管理 Linux 内核开发而开发的开放源码版本控制软件。

Git诞生于2005年：此前Linux内核社区使用BitKeeper进行版本管理，2005年双方合作终止，Linus Torvalds基于使用BitKeeper的经验，开发了全新的分布式版本控制系统Git，并制定了系统目标：

  • 速度快
  • 简单的设计
  • 对非线性开发模式的强力支持（允许成千上万个并行开发的分支）
  • 完全分布式
  • 有能力高效管理类似 Linux 内核一样的超大规模项目（速度和数据量）

核心命令说明：

  1. clone（克隆）：从远程仓库中克隆代码到本地仓库
  2. checkout（检出）：从本地仓库中检出一个仓库分支然后进行修订
  3. add（添加）：在提交前先将代码提交到暂存区
  4. commit（提交）：提交到本地仓库，本地仓库中保存修改的各个历史版本
  5. fetch（抓取）：从远程库抓取到本地仓库，不进行任何的合并动作，日常使用较少
  6. pull（拉取）：从远程库拉到本地库，自动进行合并(merge)并更新到工作区，等同于fetch+merge
  7. push（推送）：修改完成后，将代码推送到远程仓库，和团队成员共享

3.4.9、开发中分支使用原则与流程

实际项目开发中，通常会划分不同职能的分支，遵循标准的分支管理流程：

  • master（生产）分支：线上主分支，对应正式运行的代码，中小规模项目直接作为线上分支。
  • develop（开发）分支：从master创建，作为开发团队的主开发分支，日常开发都在此分支进行；阶段开发完成后，合并到master分支准备上线。
  • feature/xxxx 功能分支：从develop创建，用于并行开发不同期上线的功能，功能开发完成后合并回develop分支。
  • hotfix/xxxx 修复分支：从master派生，用于紧急修复线上Bug，修复完成后需要同时合并到master、develop等分支。

除此之外还有test（测试）分支、pre（预上线）分支等，根据项目规模按需使用。

4、 Git远程仓库

4.1、 常用的代码托管服务

Git仓库分为本地仓库和远程仓库，我们可以借助第三方代码托管平台搭建远程仓库，主流平台有：

  • GitHub：地址 https://github.com/ ，全球最大的开源代码托管平台，仅支持Git格式，开源项目免费使用。
  • 码云（Gitee）：地址 https://gitee.com/ ，国内的代码托管平台，服务器在国内，访问速度更快，支持私有仓库。
  • GitLab：地址 https://about.gitlab.com/ ，开源的仓库管理系统，通常用于企业、学校内部搭建私有Git私服。

4.5.7、解决远程合并冲突

多人协作时，A、B两人同时修改了同一个文件的同一行，A先推送到远程，B推送时会失败。

解决流程：

  1. B先执行git pull拉取远程最新代码
  2. 本地自动合并失败，产生冲突
  3. B手动解决本地冲突，add后commit提交
  4. 再次执行git push推送到远程

远程冲突的解决方式和本地分支冲突完全一致。

5.2.11、解决冲突

执行pull或merge操作发生冲突时，IDEA会弹出冲突解决窗口：

  • Accept Theirs：保留对方分支的修改，丢弃本地修改
  • 手动编辑：在代码区直接修改，保留最终需要的内容

冲突解决完成后，文件会自动加入暂存区，提交后即可完成合并，后续正常推送即可。

5.3、IDEA常用GIT操作入口

  1. 顶部快捷工具栏：更新（pull）、提交（commit）、推送（push）、分支切换、日志查看，满足日常开发需求
  2. 右键Git菜单：在项目文件/目录上右键选择Git，可执行add、revert、compare with history等操作
  3. 顶部VCS菜单：包含所有Git操作的完整入口

5.4、团队协作综合场景

练习场景：组长搭建项目仓库，组员协同开发

  1. 组长基于本地项目创建本地仓库，创建远程仓库，将项目推送到远程
  2. 所有组员从远程仓库克隆项目到本地，获得完整的开发副本
  3. 组员A修改代码，提交本地后推送到远程；组员B执行pull即可获取最新代码
  4. 组员A和组员B同时修改同一文件同一行，先后推送时后者会失败
    1. 解决：后推送的组员先pull拉取最新代码，本地解决冲突，提交后再推送

附：几条铁令

  1. 切换分支前，先提交本地的修改
  2. 代码及时提交，提交过的代码就不会丢失
  3. 遇到任何问题都不要删除.git目录，第一时间找协助人员

打开文件资源管理器，点击顶部「查看」选项卡，勾选「隐藏的项目」即可显示。

  1. 打开 File → Settings → Tools → Terminal
  2. 将 Shell path 修改为Git Bash的路径，例如C:\Program Files\Git\bin\bash.exe
  3. 点击OK保存，重启Terminal即可使用Git Bash
